# PII 検出設定ガイド (PII Detection Configuration)

Misogi の個人情報（Personally Identifiable Information）検出エンジンの設定方法について、
日本の法規制（マイナンバー法・APPI）に準拠した形で解説します。

**対象読者**: セキュリティ管理者、コンプライアンス担当者  
**法的背景**: マイナンバー法（行政手続における特定の個人を識別するための番号の利用等に関する法律）、
個人情報保護法（APPI）

---

## 1. PII 検出エンジンの概要

### 1-1. 仕組み

```
入力ファイル (.pdf/.docx/.txt/.jtd→pdf)
    │
    ▼ テキスト抽出
┌─────────────────────┐
│   テキストコンテンツ     │ ← PDF テキスト抽出 / OOXML XML 解析
└──────────┬──────────┘
           │
           ▼
┌─────────────────────────────┐
│      RegexPIIDetector        │ ← 正規表現ベースのスキャンエンジン
│  ┌───────────────────────┐  │
│  │ Rule: my_number        │  │ ← \b\d{12}\b
│  │ Rule: email            │  │ ← RFC 5322 パターン
│  │ Rule: credit_card      │  │ ← 16桁カード番号
│  │ Rule: phone_jp         │  │ ← 日本電話番号
│  │ ...                     │  │
│  └───────────────────────┘  │
└──────────┬──────────────────┘
           │
           ▼
    ┌──────┴──────┐
    │  検出結果      │
    │  found: bool │
    │  action:      │ ← Block / Mask / AlertOnly
    │  matches[]    │
    └─────────────┘
```

### 1-2. サポートする PII タイプ（デフォルトルール）

| ルール名 | 正規表現パターン | 検出対象 | デフォルトアクション | 法規制対応 |
|---------|----------------|---------|-------------------|----------|
| `my_number` | `\b\d{12}\b` | マイナンバー（12桁） | **Mask** | 📋 マイナンバー法 |
| `email` | RFC 5322 簡易パターン | メールアドレス | AlertOnly | 📋 APPI |
| `ip_address_v4` | IPv4 dotted-decimal | IP アドレス | AlertOnly | — |
| `credit_card` | `\b\d{4}[\s-]?\d{4}[\s-]?\d{4}[\s-]?\d{4}\b` | クレジットカード番号 | **Mask** | 📋 PCI-DSS |
| `phone_jp` | 日本国内電話番号パターン | 電話番号 | AlertOnly | 📋 APPI |
| `postal_code_jp` | `\b\d{3}-\d{4}\b` | 郵便番号 | AlertOnly | 📋 APPI |
| `drivers_license` | `\b\d{10,12}\b` | 運転免許証番号 | **Mask** | 📋 道路交通法 |

## 2. 基本設定

### 2-1: `[pii]` セクション

```toml
[pii]
enabled = true
action_on_detect = "mask"
```

| フィールド | 型 | デフォルト | 必須 | 説明 |
|----------|-----|---------|------|------|
| `enabled` | Boolean | `true` | ✅ | PII 検出エンジンの有効/無効 |
| `action_on_detect` | String | `"mask"` | ✅ | **デフォルト**のアクション。各ルールで個別指定も可能 |

### 2-2: アクションの種類と選択基準

| アクション | 挙動 | 使用すべき場面 | リスク |
|----------|------|--------------|--------|
| `"block"` | ファイル転送を**完全拒否** | 機密性が最優先される環境 | 誤検出で業務停止の恐れ |
| `"mask"` | 検出箇所を**マスキング**して転送 | **バランス重視（推奨）** | マスキング後も一部情報が推測可能な場合あり |
| `"alert_only"` | **ログ記録のみ**で転送継続 | 運用開始初期 / 監査目的 | 個人情報漏洩リスクあり |

### 2-3: セキュリティレベル別推奨設定

#### レベル 1: 監視のみ（導入初期）

```toml
[pii]
enabled = true
action_on_detect = "alert_only"
```

→ どのファイルに PII が含まれているかを把握するフェーズ。

#### レベル 2: マスキング（通常運用）

```toml
[pii]
enabled = true
action_on_detect = "mask"
```

→ マイナンバー・クレジットカード番号等を自動マスキング。

#### レベル 3: ブロック（高セキュリティ）

```toml
[pii]
enabled = true
action_on_detect = "block"
```

→ PII を含むファイルは一切通さない。発信元への通知が必要。

## 3. マイナンバー検出の詳細設定

### 3-1: マイナンバーとは

マイナンバー（個人番号）は、日本の住民すべてに付与される 12 桁の番号です。
マイナンバー法により、その取扱いには厳格な規制が課せられています。

| 属性 | 値 |
|------|-----|
| 桁数 | 12 桁（固定）|
| 文字種 | 数字のみ (0-9) |
| 形式 | 連続した 12 桁（区切り文字なし）|
| 例 | `123456789012` |

### 3-2: デフォルト検出パターン

```
正規表現: \b\d{12}\b
説明: ワード境界で囲まれた 12 桁の数字列に一致
```

**検出例**:

| 入力テキスト | 検出結果 | マスク結果 |
|------------|---------|-----------|
| `マイナンバーは123456789012です` | ✅ 検出 | `1**********2` |
| `電話番号03-1234-5678` | ❌ 非検出 | — |
| `ID: 123-456-789-012` | ❌ 非検出（ハイフンあり）| — |
| `番号1234567890123456` | ❌ 非検出（16桁）| — |

### 3-3: 誤検出への対策

12 桁の数字列はマイナンバー以外にも存在し得るため、誤検出（False Positive）に注意が必要です。

| 誤検出パターン | 例 | 対策 |
|--------------|-----|------|
| 伝票番号 | `伝票No. 20240115001` | コンテキスト判断（将来的 ML 対応）|
| 口座番号（一部銀行）| `口座: 123456789012` | 銀行別のフォーマット確認 |
| 製品シリアル番号 | `SN: 987654321098` | `alert_only` で様子見 |

**現状の推奨運用**: 
- 導入初期は `action_on_detect = "alert_only"` で誤検出率を計測
- 誤検出が許容範囲内（< 1%）になったら `mask` へ切り替え
- 重要文書のみ `block` を適用

## 4. カスタム PII ルールの作成

### 4-1: 新しいルールの追加方法

Misogi の PII エンジンは拡張可能です。組織固有の PII パターンを追加できます。

**現在のバージョンでは**、カスタムルールはソースコードレベルで追加します。
将来のバージョンでは設定ファイルからのルール追加を予定しています。

#### カスタムルールの定義例（Rust ソースコード）

```rust
// crates/misogi-core/src/pii/custom_rules.rs
use crate::pii::{PIIRule, PIIAction};

/// 社員番号検出ルール（8桁の英数字）
pub fn employee_id_rule() -> PIIRule {
    PIIRule::new(
        "employee_id",                           // ルール名
        r"\b[A-Z]{2}\d{6}\b",                   // 正規表現: 2文字+6数字
        PIIAction::Mask,                         // アクション: マスキング
        '#',                                     // マスク文字
        "Employee ID number (e.g., AB123456)",   // 説明
    )
}

/// 健康保険被保険者番号検出
pub fn health_insurance_rule() -> PIIRule {
    PIIRule::new(
        "health_insurance_no",
        r"\b\d{4}[\s-]?\d{4}[\s-]?\d{4}\b",
        PIIAction::Block,
        'X',
        "Japanese Health Insurance number",
    )
}
```

### 4-2: 一般的なカスタムルール例

| ルール名 | 正規表現 | 日本語での用途 |
|---------|---------|-------------|
| 社員番号 | `\b[A-Z]\d{6}\b` | 社内人事システム |
| 健康保険番号 | `\b\d{11}\b` | 社会保険関連 |
| パスポート番号 | `\b[A-Z]{2}\d{7}\b` | 旅費精算書類 |
| 顧客コード | `\bCUST-\d{6}\b` | CRM システム連携 |
| 予算コード | `\bY\d{4}-[A-Z]{2}\d{3}\b` | 予算管理システム |

## 5. エンコーディングと PII 検出

### 5-1: サポートするエンコーディング

日本のレガシーシステムで作成されたファイルは、様々な文字エンコーディングを使用しています。
Misogi は以下のエンコーディングを自動検出・変換して PII スキャンを行います。

| エンコーディング | IANA 名 | 典型的な出典 | 自動検出信頼度 |
|---------------|---------|-----------|-------------|
| UTF-8 | `UTF-8` | 現代 Web/API システム | ⭐⭐⭐ 最高 (BOM あれば 1.0) |
| Shift-JIS (CP932) | `Shift_JIS` | Windows レガシーアプリ | ⭐⭐ 高 (周波数解析) |
| Windows-31J | `Windows-31J` | Microsoft 固有 SJIS | ⭐⭐ 高 |
| EUC-JP | `EUC-JP` | Unix/Linux レガシー | ⭐⭐ 中 (周波数解析) |
| ISO-2022-JP (JIS) | `ISO-2022-JP` | メール (JIS コード) | ⭐⭐⭐ 高 (エスケープ配列) |
| UTF-16 LE/BE | `UTF-16LE` | Windows API / Office | ⭐⭐⭐ 最高 (BOM) |

### 5-2: エンコーディング自動検出アルゴリズム

```
検出優先順位（信頼度が高い順）:

1. BOM (Byte Order Mark) 検出
   ┌──────────┬──────────┬──────────────┐
   │ バイト列  │ エンコード │ 信頼度       │
   ├──────────┼──────────┼──────────────┤
   │ EF BB BF │ UTF-8    │ 1.0 (確定)   │
   │ FE FF    │ UTF-16BE │ 1.0 (確定)   │
   │ FF FE    │ UTF-16LE │ 1.0 (確定)   │
   └──────────┴──────────┴──────────────┘

2. ISO-2022-JP エスケープ配列検出
   ESC $ B (0x1B 24 42) → JIS X 0208-1983 (95% 信頼)

3. UTF-8 有効性チェック
   全バイトが有効な UTF-8 シーケンス → UTF-9 (90% 信頼)

4. Shift-JIS vs EUC-JP バイト周波数解析
   ・SJIS 1st byte: 0x81-0x9F, 0xE0-0xEF
   ・EUC-JP 1st byte: 0xA1-0xFE
   → 各範囲の出現頻度で判定 (60-90% 信頼)

5. フォールバック
   → 設定された fallback_encoding (デフォルト: Shift_JIS)
```

### 5-3: エンコーディング設定

```toml
# 将来の拡張予定（現在はコード内ハードコード）
[pii.encoding]
fallback = "Shift_JIS"     # 自動検出失敗時のデフォルト
pdf_font_action = "preserve" # 不明フォームの取り扱い
```

**`pdf_font_action` のオプション**:

| 値 | 動作 | セキュリティ | 可読性 |
|---|------|------------|--------|
| `"preserve"` | そのまま保持 | ⚠️ フォント攻撃ベクトル残存可能 | ✅ 最高 |
| `"strip"` | 不明フォント参照を削除 | ✅ 最も安全 | ❌ 文字化けの可能性 |
| `"replace"` | 安全なフォントに置換 | ✅ 安全 | △ 代替フォーマットに依存 |

## 6. Block / Mask / Alert の詳細動作

### 6-1: Mask（マスキング）の仕組み

```
原文:  "マイナンバーは123456789012です"
               └──────┘
               検出!
                    ↓
マスク: "マイナンバーは1**********2です"

ルール:
・3 文字以上の場合: 先頭 1 文字 + 中間マスク + 末尾 1 文字を保持
・2 文字以下: 全てマスク文字に置換
・マスク文字: ルールごとに設定可能（*, X, ●, # 等）
```

**各ルールのマスク例**:

| ルール | 原文 | マスク結果 | マスク文字 |
|-------|------|-----------|-----------|
| my_number | `123456789012` | `1**********2` | `*` |
| credit_card | `4111 1111 1111 1111` | `4XXXXXXXXXXXXXXXXX1` | `X` |
| email | `user@example.com` | `************` | `*` |
| drivers_license | `A123456789012` | `A###########0` | `#` |

### 6-2: 複数ルールが同時にマッチした場合

アクションの優先順位: **Block > Mask > AlertOnly**

```
例: ファイルに「マイナンバー」と「クレジットカード」が両方含まれる場合

my_number ルール → action: Mask
credit_card ルール → action: Mask
                ↓ 最も厳しいアクションを採用
        最終アクション = Mask （両方とも Mask なので）

例: マイナンバー(Block) + メールアドレス(AlertOnly)

my_number ルール → action: Block    ← 厳しい
email ルール     → action: AlertOnly
                ↓
        最終アクション = Block （Block が優先）
```

## 7. PII 検出の限界と注意点

### 7-1: 技術的限界

| 制限 | 説明 | 将来の対応 |
|------|------|----------|
| **画像内の PII** | スキャン画像や写真に写ったマイナンバーは検出不可 | OCR 連携（開発中）|
| **暗号化コンテンツ** | パスワード保護ファイル内は検索不可 | 復号モジュール（検討中）|
| **分割された番号** | `123-456-789-012` のように区切られた番号 | パターン拡張（容易）|
| **偽装された PII** | `私のマイナンバーは「一二三四五六七八九〇一二」です` | NLP ベース検出（検討中）|
| **音声/動画** | 音声ファイルや動画内の個人情報 | 音声認識連携（長期的）|

### 7-2: 運用上の推奨事項

1. **定期的な誤検出レビュー**: 月に 1 回、`alert_only` モードでの検出結果をレビュー
2. **閾値調整**: 組織の実態に合わせてルールの厳しさを調整
3. **多層防御**: PII 検出だけでなく、教育・プロセス・技術的なアクセス制御を組み合わせる
4. **ログ監視**: PII 検出ログの異常増加を早期に検知するアラート設定
